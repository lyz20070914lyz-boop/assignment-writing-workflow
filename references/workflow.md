# 完整工作流程

## 1. Intake

1. 列出已读取和无法读取的文件。
2. 提取题目、交付物、任务类型、字数、rubric、HD 描述、来源要求、格式、截止日期、提交方式和 AI 规则。
3. 缺失内容标记为 `文件未说明`。
4. 询问是否创建文档1、文档2；确认名称、位置、新建或更新、内容和格式修改权限。
5. 创建状态记录：当前阶段、任务类型、已确认规则、待确认项、写作逻辑版本、已选文献、下一步。

## 2. 中文任务说明

输出：

1. 这项作业要做什么
2. 必须提交什么
3. HD需要出现哪些可观察证据
4. 容易失分的地方
5. 文件未说明
6. 必须确认的问题

把 rubric 原文与解释分开。没有 rubric 时，不生成虚构的 HD 标准；通用建议标记为 `建议`。

## 3. 范例与写作逻辑

在首次大纲或正文前：

1. 分析教师范例。
2. 查找同任务类型、相近学科和相近目的的外部可靠范例。
3. 比较结构、中心任务、段落功能、证据使用、限制处理、引用和与 rubric 的关系。
4. 标记 `可采用`、`仅供参考`、`不适用`、`无法核实`。
5. 生成 `本次任务写作逻辑`：任务目的、section 顺序、每部分功能、证据规则、整合方式、限制处理、写作顺序、范例依据和不确定性。

不复制范例句子、内容、个人经历或专有结构。rubric 与范例冲突时，以 rubric 为准。

## 4. 研究和大纲

1. 把写作逻辑拆成论点或分析任务。
2. 为每项确定需要的证据。
3. 搜索、验证并筛选文献。
4. 保留淘汰原因。
5. 建立论点—证据—文献—rubric 对应关系。
6. 生成大纲；每项写明功能、论点、来源、与总任务的联系和目标字数。

## 5. 写作和修改

1. 按任务模式确定的顺序写，不默认 introduction-first。
2. 完整流程中一次完成一个段落或 section。
3. 检查该部分的功能、证据、引用和与总任务的联系。
4. 用户确认后再进入下一部分，并在获准时更新文档1。
5. 中文稿完成后检查题目、rubric、论证、结构、证据、引用和字数。
6. 用户自行修改后，按需替换专业术语。
7. 用户采用 Apple Translate 时暂停等待英文稿。
8. 对英文执行低改动检查。
9. 从最新正文生成 APA 7 References。

## 6. 文档边界

文档1用于任务说明、范例比较、写作逻辑、文献记录、大纲、中文稿和修改记录。保留状态记录以便跨会话恢复。

文档2只包含用户确认的提交内容。格式只依据课程文件；缺失项标记 `文件未说明` 并询问。

不得覆盖原文件。创建或更新 DOCX 后检查文字完整性和页面效果。

## 7. 最终检查

按顺序检查：题目与 rubric、中心任务、section 功能、证据与引用、正文与 References 双向匹配、字数、格式、提交组件、未解决占位符。

问题分为：`提交前必须修复`、`有时间再修复`、`已核对`。存在必要缺口时，不得声称可以提交。
