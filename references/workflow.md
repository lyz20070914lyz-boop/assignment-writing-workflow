# 完整工作流程

## 1. Intake 与状态

1. 列出已读取和无法读取的课程文件。
2. 提取题目、交付物、任务类型、字数、rubric、HD 描述、来源要求、格式、截止日期、提交方式和 AI 规则。
3. 缺失内容标记为 `文件未说明`；只询问会阻塞当前阶段的问题。
4. 确认是否创建文档1、文档2，以及名称、位置、新建或更新、内容和格式修改权限。
5. 记录：当前阶段、执行层级、任务类型、中心控制项、已确认规则、rubric 映射版本、写作逻辑版本、来源状态、待确认项和下一步。

AI 规则触发 `受限辅助模式` 时，记录允许和禁止的范围，并只推进允许的阶段。

## 2. 任务与 Rubric

输出中文任务说明：作业要回答什么、必须提交什么、rubric 的可观察表现、每项标准对应的 section 任务、容易失分处、文件未说明和必须确认的问题。

读取 argument-quality，建立 rubric 映射和中心控制项。没有 rubric 时，不生成 HD 标准；把通用判断标记为 `建议`。

阶段门槛：题目中的任务动词、范围、交付物和中心控制项已经明确，或剩余假设已显式记录。

## 3. 范例与本次写作逻辑

1. 分析教师范例的结构、深度、段落功能、证据用法、限制处理和格式。
2. 查找同任务类型、相近学科和相近目的的外部可靠范例。
3. 标记 `可采用`、`仅供参考`、`不适用` 或 `无法核实`，并比较共同点和差异。
4. 课程文件和 rubric 高于任何范例；不复制句子、内容、经历或特殊表达。
5. 生成 `本次任务写作逻辑`：中心任务、section 顺序、各部分功能、证据规则、整合方式、限制处理、写作顺序、范例依据和不确定性。

阶段门槛：写作逻辑能够解释每个 section 如何完成题目和 rubric；没有可靠范例时已明确说明。

## 4. 研究与证据

1. 把中心控制项和分论点转成证据需求。
2. 按 evidence 的规则搜索、核验、筛选来源并记录淘汰原因。
3. 建立句子或主张级的来源支持记录；区分直接、条件性、间接、背景和不支持。
4. 不得用未核实来源填补大纲。

阶段门槛：每个核心主张已有足够的已验证支持，或明确标出证据缺口和处理方案。

## 5. 大纲

按 argument-quality 建立完整论证链。每个主要 section 写明功能、主张或分析任务、证据、推理、限制或替代解释、对中心任务的贡献、rubric 对应和目标字数。

检查总字数分配、section 依赖关系、重复论点和无归属 rubric 项。将大纲和已选文献写入文档1前取得许可。

阶段门槛：删除任一主要 section 都会造成明确的题目、论证或 rubric 缺口；证据不足处不得进入成稿写作。

## 6. 写作与局部检查

1. 按任务模式确定的顺序写，不默认 introduction-first。
2. 一次处理一个 section；仅在方向会影响后文时等待确认。
3. 每部分完成后检查功能、主张、证据、推理、限制、连接和 rubric。
4. 在获准时更新文档1，保留来源状态和未解决项。
5. 中文稿完成后检查中心控制项是否仍与实际论证一致。
6. 用户自行修改后，按 revision 进行术语替换、Apple Translate 交接或英文低改动。

## 7. 全文审计

依次检查：题目与 rubric、中心控制项、section 贡献、跨段逻辑、证据支持强度、概念一致性、限制与替代解释、正文引用与 References 双向匹配、字数和课程格式。

执行 argument-quality 的全文反向审阅。问题按 `提交前必须修复`、`优先修改`、`可选` 分类。不得把风格偏好列为必须修改。

## 8. 文档与提交

文档1保存任务说明、rubric 映射、范例比较、写作逻辑、来源审计、大纲、中文稿和修改记录。

文档2只保存用户确认的提交内容。格式只依据课程文件；缺失项标记 `文件未说明`。不得覆盖原文件。创建或更新 DOCX 后检查文字完整性和页面效果。

最终核对提交组件、字数、格式、未解决占位符和 AI 使用要求。存在必要缺口时，不得声称可以提交。
